# CI/CD 파이프라인 중앙화를 위한 재사용 워크플로우(Reusable Workflows) 도입 기획안

## 1. 도입 배경 및 필요성
현재 Dental 시스템은 15개의 독립된 레포지토리로 분할되어, GitHub Actions의 가상 머신 자원을 무료로 극대화하여 활용하는 **효율적인 분산 병렬 컴퓨팅 아키텍처**를 채택하고 있습니다. (레포지토리당 2코어/7GB RAM 무료 할당)
하지만 이를 구동하는 `.github/workflows/test.yml` 스크립트가 15곳에 중복으로 복사되어 있어, CI/CD 정책(예: 파이썬 버전 변경, 의존성 패키지 추가)이 변경될 때마다 15개 레포지토리에 동일한 패치를 커밋하고 푸시해야 하는 **유지보수의 번거로움과 동기화 누락 위험**이 존재합니다.

이를 해결하기 위해, 현재의 클라우드 분산 처리 이점은 100% 유지하면서 CI 스크립트의 코드 중복만 획기적으로 제거할 수 있는 **'GitHub Actions Reusable Workflows(재사용 워크플로우)'**의 도입이 필요합니다.

## 2. 아키텍처 개요
- **중앙(Caller):** `Dental_000` 레포지토리에 단일 마스터 워크플로우(`reusable-test.yml`)를 정의합니다. (`on: workflow_call` 속성 사용)
- **종속(Callee):** 나머지 14개의 개별 모듈(001~014)은 자신의 `test.yml`에서 실행 단계를 길게 나열하는 대신, 단 1줄의 `uses:` 구문을 사용하여 000 모듈의 마스터 워크플로우를 원격 호출합니다.

```yaml
# 개별 레포지토리(001~014)의 test.yml 예시
name: E2E Pipeline
on: [push, pull_request]

jobs:
  call-central-workflow:
    uses: HyunchanAn/Dental_000/.github/workflows/reusable-test.yml@master
    with:
      target_module: "tests_001"
```

## 3. 기대 효과
1. **단일 진실 공급원(SSOT) 확보:** CI/CD 파이프라인 로직이 `Dental_000` 한 곳에서만 관리되므로 수정 및 배포가 즉각적이고 안전합니다.
2. **분산 병렬 컴퓨팅 유지:** 코드는 중앙에서 가져오지만, 파이프라인의 물리적 실행은 여전히 개별 레포지토리에 할당된 15개의 깃허브 원격 서버에서 병렬로 이루어지므로 OOM(메모리 초과) 위험이 없고 빠릅니다.
3. **확장성(Scalability):** 추후 015, 016 등 새로운 의료 모듈이 추가되더라도 YAML 복사 붙여넣기 없이 즉시 파이프라인 연동이 가능합니다.

## 4. 단계별 실행 계획 (Action Plan)
- **Phase 1 (준비 단계):** `Dental_000`의 `.github/workflows` 디렉토리에 `workflow_call` 트리거를 포함하는 `reusable-test.yml` 생성. (환경 변수 및 의존성 설치 일원화)
- **Phase 2 (시범 적용):** 영향도가 적은 모듈 1개(예: `Dental_001`)의 기존 `test.yml`을 삭제하고, 재사용 워크플로우를 호출하도록 변경한 뒤 작동 무결성 테스트.
- **Phase 3 (전면 도입):** 001의 검증이 끝나면 스크립트 봇을 이용해 나머지 13개 모듈의 `test.yml`을 일괄 교체 및 일괄 푸시.
- **Phase 4 (모니터링):** 15개 전체 모듈이 `Dental_000`의 마스터 워크플로우를 참조하여 정상적으로 `completed success`를 띄우는지 대시보드 검증.

## 5. 결론
본 기획은 하드웨어 병목을 극복한 현재의 우수한 분산 아키텍처를 훼손하지 않고, 유지보수의 파편화 문제만을 외과 수술처럼 정교하게 도려내는 가장 **비용 효율적이고 완벽한 인프라 고도화 전략**입니다. 향후 리소스 여유가 생겼을 때 우선순위로 추진하는 것을 권장합니다.
